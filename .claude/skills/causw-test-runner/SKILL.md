# CAUSW Test Runner

테스트를 실행하고, 실패한 테스트 코드를 수정하고, 다시 실행한다. 최대 5회 반복 후 결과를 보고한다.

## 핵심 규칙

- **테스트 코드만 수정한다.** `src/test/` 하위 파일만 편집 가능하다.
- `src/main/` 프로덕션 코드는 절대 수정하지 않는다. 프로덕션 코드에 버그가 있다고 판단되면 사용자에게 보고하고 멈춘다.
- 테스트 코드 작성 컨벤션은 `causw-test-guide` 스킬을 참고한다.

---

## 실행 절차

### 1단계: 테스트 실행

프로젝트 루트에서 실행한다:

```bash
./gradlew :app-main:test --continue 2>&1
```

`--continue` 옵션으로 실패가 있어도 전체 테스트를 모두 실행한다.

### 2단계: 결과 분석

테스트 실행 후 XML 결과 파일을 파싱한다:

```bash
# 실패한 테스트 목록 확인
grep -r '<failure\|<error' app-main/build/test-results/test/ -l
```

각 실패 XML 파일에서 다음 정보를 추출한다:
- 테스트 클래스명 (classname)
- 테스트 메서드명 (name)
- 실패 메시지 (failure/error 태그 내용)
- 스택 트레이스

모든 실패를 한꺼번에 모은 뒤 수정에 들어간다.

### 3단계: 실패 원인 분류 및 수정

실패 유형별로 접근한다:

| 유형 | 증상 | 대응 |
|------|------|------|
| **Mock 설정 누락** | `WantedButNotInvoked`, `UnnecessaryStubbingException` | given/then mock 설정 추가 또는 제거 |
| **Assertion 불일치** | `AssertionError`, expected vs actual 불일치 | 기대값 수정 또는 테스트 로직 재검토 |
| **컴파일 오류** | 메서드/필드 없음, 타입 불일치 | import 추가, 메서드 시그니처 맞춤 |
| **NPE / 초기화 누락** | `NullPointerException` | Fixture 또는 mock 반환값 설정 |
| **프로덕션 코드 변경 반영 필요** | 리팩토링 후 테스트가 구버전 API 사용 | 테스트를 새 API에 맞게 업데이트 |

**수정 시 주의사항:**
- 실패 이유를 이해한 뒤 수정한다. 증상만 보고 assert를 억지로 맞추지 않는다.
- `@Disabled`나 `// TODO` 처리로 테스트를 회피하지 않는다.
- 실패의 원인이 프로덕션 코드 버그라고 판단되면 수정을 멈추고 사용자에게 보고한다.

### 4단계: 재실행 및 반복

수정 후 다시 1단계로 돌아간다. **최대 5회 반복**한다.

---

## 반복 루프 흐름

```
실행 → 결과 분석
  ├─ 전부 성공 → 완료 보고 (종료)
  ├─ 실패 존재 & 반복 < 5회 → 테스트 코드 수정 → 재실행
  ├─ 프로덕션 버그 발견 → 사용자 보고 후 종료
  └─ 5회 도달 & 여전히 실패 → 남은 실패 목록과 원인 보고 후 종료
```

### 성공 시

```
✅ 테스트 전체 통과
- 총 N회 실행
- 수정한 파일: [파일 목록]
- 수정 내용 요약: [변경 사항 간략 설명]
```

### 실패 잔존 시 (5회 초과 또는 프로덕션 버그)

```
❌ 테스트 미완료 (N회 실행)
- 남은 실패 테스트: [클래스명#메서드명 목록]
- 원인: [각 실패 이유]
- 권장 조치: [사용자가 직접 처리해야 할 내용]
```
